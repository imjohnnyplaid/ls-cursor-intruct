# PROJECT STRUCTURE – Language Skull

This document defines the canonical folder and file organization for the Language Skull iOS codebase.  
All code generated by Cursor **must** follow this structure. Do not create new top-level folders without updating this document.

## 1. Philosophy

- **Feature-first** for user-facing areas (Home, StudySession, Calendar, etc.)
- **Core layers** for shared architecture, data access, and business logic
- **Clear separation** between UI, business logic, and persistence
- Easy to navigate for both humans and Cursor
- Supports the phased development plan in `docs/13_INCREMENTAL_BUILD_PLAN.md`
- Admin mode is feature-flagged and should remain easy to completely remove before App Store submission

## 2. Root Folder Structure

```
LanguageSkull/
├── App/                          # App entry point + configuration
├── Features/                     # All user-facing features (feature-first)
├── Core/                         # Cross-cutting architecture, data, and services
├── Shared/                       # Reusable UI components, theme, utilities
├── Resources/                    # Assets + bundled content (JSON, Markdown)
├── Preview Content/              # SwiftUI preview data factories
├── Tests/                        # Unit + UI tests
└── LanguageSkull.xcodeproj
```

## 3. Detailed Breakdown

### App/
Contains only the absolute minimum to launch the app.

```
App/
├── LanguageSkullApp.swift
├── AppDelegate.swift                 (only if needed for Firebase / notifications)
├── FeatureFlags.swift                (central place for @AppStorage / environment flags)
└── Theme.swift                       (color palette, fonts, materials — see docs/07)
```

### Features/
Each feature lives in its own folder with a clear internal structure.  
Use this pattern for every major screen/flow.

```
Features/
├── Onboarding/
│   ├── OnboardingView.swift
│   ├── OnboardingViewModel.swift
│   └── OnboardingCoordinator.swift   (if navigation logic grows)
├── Home/
│   ├── HomeView.swift
│   ├── HomeViewModel.swift
│   └── Components/
├── StudySession/
│   ├── StudySessionView.swift
│   ├── StudySessionViewModel.swift
│   ├── FlashcardView.swift
│   ├── TwoColumnListView.swift
│   └── GrammarParagraphView.swift
├── Calendar/
│   ├── CalendarView.swift
│   └── CalendarViewModel.swift
├── Profile/
│   ├── ProfileView.swift
│   └── ProfileViewModel.swift
└── Admin/
    ├── AdminTabView.swift
    ├── ContentImportView.swift
    ├── StudyPlanEditorView.swift
    └── AdminViewModel.swift
```

**Rule**: Business logic lives in the `*ViewModel.swift`. Views contain **only** UI and binding code.

### Core/
This is where the architecture lives.

```
Core/
├── Domain/
│   └── Models/                       # Pure Swift models / value types (if any)
├── Data/
│   ├── Repositories/
│   │   ├── UserRepository.swift
│   │   ├── ContentRepository.swift
│   │   └── ProgressRepository.swift
│   ├── SwiftData/
│   │   └── Models/                   # All @Model classes (from docs/04)
│   └── Content/
│       └── ContentSeeder.swift
├── Services/
│   ├── StudyPlanEngine.swift         # Critical — owns all session assembly logic (docs/06)
│   ├── SubscriptionManager.swift
│   └── AnalyticsService.swift
└── Utilities/
    └── Date+Extensions.swift
```

**Important**:
- `StudyPlanEngine` is the single source of truth for Morning/Evening session assembly and D-1 / D-3 logic.
- All repositories are **actors** when they perform I/O.
- SwiftData models live only under `Core/Data/SwiftData/Models/`.

Shared/

│   │   └── Materials.swift

│       ├── SkullButton.swift
│       ├── ProgressRing.swift
│       └── AvatarView.swift
└── Extensions/

### Resources/

```
Resources/
├── Assets.xcassets
├── Content/                          # Bundled seed data
│   ├── spanish_words.json
│   ├── spanish_phrases.json

```

### Preview Content/
Contains factories and sample data used **only** in `#Preview` macros.  
Never imported in production code.

```
Preview Content/
├── PreviewData.swift
├── SampleWords.swift
└── SampleStudyPlans.swift
```

## 4. Naming Conventions

- Views: `FeatureNameView.swift`
- ViewModels: `FeatureNameViewModel.swift`
- Repositories: `DomainRepository.swift` (e.g. `ProgressRepository`)
- Services / Engines: `StudyPlanEngine.swift`, `SubscriptionManager.swift`
- SwiftData models: PascalCase matching the entity (`UserProfile.swift`, `Word.swift`)

## 5. Cursor Rules for This Structure

When generating code, Cursor **must**:

- Place new SwiftUI screens under the appropriate `Features/` subfolder.
- Place all data access logic in `Core/Data/Repositories/`.
- Place business logic that is not UI-specific in `Core/Services/` (especially anything related to study plan assembly).
- Never put SwiftData `@Model` classes anywhere except `Core/Data/SwiftData/Models/`.
- Keep Admin-related code inside `Features/Admin/` and guard it with the admin feature flag.
- Update this document if a new top-level folder is truly required.
